## 2. 개정안 — 현재 수행력 점수 (Cross-sectional)

### 2-1. 메카닉 영역 종합 — 카테고리 가중치 도입

**현재 (v31.12)**: `Mechanics_Score = 단순 평균(C1~C5)`
**개정**: 효과 크기에 비례한 가중 평균

| 카테고리 | 현 가중치 | **개정 가중치** | 근거 |
|---|---:|---:|---|
| C1_LowerBodyDrive | 1.0 (균등) | **1.0** | cross d=+0.94 (Large) |
| C2_FrontLegBlock | 1.0 | **0.8** | cross d=+0.71 (Medium) |
| C3_SeparationFormation | 1.0 | **1.0** | cross d=+0.83 (Large, Foundation 포함) |
| **C4_TrunkAcceleration** | 1.0 | **1.5** | ★ cross d=+1.25 (단일 최대) |
| **C5_UpperBodyTransfer** | 1.0 | **0.5** | d=+0.09, MER 천장 효과 |

→ 합 = 4.8. C4 비중 31%, C5 비중 10%.

**적용 위치 (코드)**: `BBL_신규선수_리포트.html` 내 4226~4232 라인 부근, `Mechanics_Score` 계산 식 수정.
신규 상수 `MECHANICS_AREA_WEIGHTS` 추가 후 가중평균 함수 호출.

### 2-2. C4_TrunkAcceleration 내부 변수 가중치

**현재 → 개정**:

| 변수 | 현 가중치 | **개정** | 근거 |
|---|---:|---:|---|
| **trunk_flex_vel_max** | 0.40 | **1.00** | ★ 단일 최강 변수 (long d=+0.96, cross d=+0.89) |
| max_trunk_twist_vel_dps | 0.60 | **0.80** | cross d=+0.90 |
| pelvis_to_trunk_lag_ms | 0.80 | 0.80 | 사용자 통찰 유지 |
| max_pelvis_rot_vel_dps | 0.40 | 0.40 | 유지 |
| torso_side_bend_at_mer | 0.60 | **0.40** | 분석에서 영향 없음 |

**적용 위치**: `CATEGORY_WEIGHTS.C4_TrunkAcceleration` (4112~4118 라인 부근).

### 2-3. C5_UpperBodyTransfer 내부

부상 위험 모니터링 변수로서 **변수 가중치는 유지**, 카테고리 가중치만 0.5로 축소(2-1). MER이 코호트 평균 ~190°로 천장 효과가 명확하므로 종합 점수에 미치는 영향만 줄임.

### 2-4. F1_Strength — 악력 추가 (Phase 2 표시)

**문제**: 분석에서 Grip d=+1.18 (Large) 인데, 현재 F1_Strength에는 IMTP/체중만 포함되어 있음. `Grip Strength`가 var_polarity에는 등록되어 있으나 var_distributions에 분포가 없어 점수 산출 불가.

**조치**:
- **Phase 1 (이번 패치)**: 코드만 준비 — F1_Strength 카테고리에 `Grip / Weight[KG]` 항목을 추가하되, 분포 데이터 없으면 자동 skip.
- **Phase 2 (코호트 재학습 시)**: cohort_v29.js 재생성 시 Grip Strength 분포 포함하도록 `data_pipeline` 명세에 추가 (이 문서 §6 참고).

### 2-5. 개정 후 효과 시뮬레이션 (예시)

가상 선수 A — H2 횡단 분석에서 "느린 그룹" 평균값 입력:
- C1: 50, C2: 50, C3: 50, C4: 31.7, C5: 60

**현재(v31.12) Mechanics_Score** = (50+50+50+31.7+60)/5 = **48.3점**
**개정 Mechanics_Score** = (1.0×50 + 0.8×50 + 1.0×50 + 1.5×31.7 + 0.5×60) / 4.8 = **45.0점**

→ C4가 약점이고 C5가 좋아 보이는 위장 효과를 줄여, **약점 진단이 명확**해짐.

## 3. 개정안 — 향상 잠재력 점수 (Improvement Potential Score, IPS)

### 3-1. 정의

H1↔H2 두 시점 데이터가 모두 있을 때만 산출. **종단 분석에서 효과 큰 변수의 변화량을 z-score로 정규화한 가중 합**.

### 3-2. 산식 (Phase 1)

```
IPS = 0.40 × Z(Δtrunk_flex_vel_max)
    + 0.30 × Z(ΔC4_TrunkAcceleration_score)
    + 0.20 × Z(ΔMechanics_Score)
    + 0.10 × Z(ΔC1_LowerBodyDrive_score)
```

여기서:
- `Δ변수` = H2값 − H1값
- `Z(Δ)` = (Δ − 코호트Δ평균) / 코호트Δ표준편차
- 변환: 0~100 점수로 매핑 (z=0 → 50, z=+1 → 70, z=−1 → 30, clip [0,100])

### 3-3. 코호트 Δ 분포 (cohort_v29.js 추가 필요)

새 키 `delta_distributions` 추가:
- trunk_flex_vel_max ΔH (mean, sd, n)
- C4 score ΔH
- Mechanics_Score ΔH
- C1 score ΔH

**Phase 1**: 분석 문서의 향상/정체 그룹 통계로 잠정 사용. **Phase 2**: 134명 전체 페어드 분석으로 갱신.

### 3-4. UI 표시

- "내 점수" 카드 옆에 "향상 잠재력" 카드 신설.
- IPS ≥ 70: "급성장형" / 50~70: "정상 발달" / 30~50: "주의" / <30: "정체 위험"
- ⚠ 단일 측정 선수에게는 표시하지 않음 ("H1·H2 두 시점 데이터 필요" 안내).

### 3-5. 적용 위치

- `calculateScores()` 후속에 `calculateIPS(input_h1, input_h2)` 신규 함수 추가.
- H1↔H2 비교 뷰에서만 호출.

## 4. 변경 안 하는 것 (의도된 불변)

| 항목 | 이유 |
|---|---|
| **C2 무릎 평가 임계** | 메모리: "앞다리 버팀은 무릎 각도 유지가 핵심" — 0 이상 만점, 음수만 감점 유지 |
| **C5 max_shoulder_ER 변수** | MER 자체는 부상 모니터링 변수로 가치 있음. 천장 효과는 카테고리 가중치로 처리 |
| **좌·우완 -3 km/h offset** | 메모리: "좌·우완은 같은 조건 데이터" 원칙은 임계값에 적용. 회귀 절편 offset은 회귀식 구조상 분리 필요 |
| **시퀀싱(lag) 발달 단계 컨텍스트** | 메모리: "PelvisTrunkDisconnect 발달 미성숙은 high severity 진단 X" 유지 |
| **카테고리 결측 50% 보류** | v31.2 사용자 피드백으로 도입된 신뢰성 장치 — 변경 없음 |

## 7. 한계 및 주의 (Limitations)

1. **단일 학교(고교 1학년) 코호트** — 다른 연령/수준에 일반화 시 재검증 필요
2. **6개월 단기 종단** — 1년+ 발달 패턴은 미반영
3. **Phase 2 변수 누락** — peak_x_factor, max_cog_velo 결측 시 C3·C1 점수 신뢰성↓ (현재의 50% 보류 장치로 1차 차단됨)
4. **IPS Phase 1은 잠정값** — 가중치(0.40/0.30/0.20/0.10)는 효과 크기 비율 추정, 다중회귀 재학습 후 갱신 권장
5. **MER 천장 효과는 코호트 한정** — 다른 코호트(예: 대학·프로)에서는 MER 분산이 다시 살아날 수 있음 → 카테고리 가중치는 코호트 의존적

## 8. 승인 요청 사항

다음 4가지에 대해 의견 부탁드립니다:

1. **C5 카테고리 가중치 0.5** — 더 낮춰도(예: 0.3) / 그대로 둘지
2. **C4 카테고리 가중치 1.5** — 더 강하게(예: 2.0) 줄지
3. **trunk_flex_vel_max 변수 가중치 1.00** — 0.40 → 1.00은 큰 변경, 단계적(예: 0.70 먼저)으로 갈지
4. **IPS Phase 1 도입 시점** — 이번 v32.0에 포함 vs 별도 v32.1로 분리
